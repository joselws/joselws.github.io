About

The fist working version of this project (as of this writing) was finished on March 7th 2022.

This is an ecommerce website based on auctions. You can register yourself as a user and start using the application right away. Login and logout whenever you need to.

The main functionality of this app revolves around creating items in a certain category, sharing them with the rest of the world, and bidding on other people's items, and let other people bid on yours.

With your own item, you can decide to close the auction, letting the highest bid user win your item, edit your item, delete it, or even comment it yourself.

With other people's items, you may bid on them, comment them, or add them to your personal watchlist so you can have them in sight anytime.

You may browse items by applying filters like the most populars (items with the highest activity), recent (those who were updated least), view by categories (like books or electronics), watchlist, or your active items.

More about the app

This app was deployed using Heroku.

I spent around two weeks working on the basic features of the app, and one additional month testing, adding new features, and styling the application with CSS.

If the user doesn't upload an image, a default appropriate image representing an empty image is used. Images must be at most 2MB in size. Current validation accepts only .jpg, .jpeg, and .png file formats.

Passwords are hashed automatically by Django, so there's no risk in your password being revealed.

There are many features ideas that I may add some time in the future, you can check them out on this project's Github repository README. I would honestly like to eventually turn this project into a fully working, complete commercial application.

Obstacles and how I faced them

As with any project or working on something you're not familiar with, obstacles will naturally appear as you work on features on your project. I did encounter handful of them, but I will mention the most prominent ones.

Storing user-uploaded images. When I started thinking about how I would handle the images in this app, the first thing that popped out of my mind was "wait a second, how do databases store images?" After a bit of research, I found out that they don't. Dedicated cloud servers like AWS or Azure are required for these kind of tasks. The problem is, to use AWS or Azure, I need a credit card, which I don't have, so I thought about storing user images within the folder directory. After a bit of workarounds and tweaking with Django project settings, I was able to store user-uploaded files in a dedicated folder within the project directory.

The next problem I encountered is related to the one mentioned above: Delete user-uploaded images no longer in use. One might naturally assume that if you delete the item an image belongs to, the image will also disappear, but it doesn't. This is a problem because, on a scalable app, in the long run a lot of images would end up clustering up in the folder that are not necessary anymore, so I had to figure out a way of deleting an image as soon as its respective item gets deleted as well. That's when I found out about Django signals. I learned about them, and connected a couple of those so the image is deleted along with the item, or the image is deleted when the item's image is edited and replaced.

What's described above also leads to another problem. What if there is no image to begin with? Deleting something that doesn't exist would cause an error, so I had to find a way of Asserting that an image exists. This lead to studying the Python "os" module and using os.path.exists() method. As its name tells, this method let's you know whether a path exists.

A combination of everything described above lead to pretty interesting things I could do with Djagno Signals: handling empty user image inputs for a default, handling the popularity feature, handling the updated_at attribute, etc.

What I learned

There are plenty of things I learned through this experience. The most prominent is now my overall growth as a developer. Although the app is by itself quite simple - nothing too complex - I had to apply many of the web development practices fundamentals, which led to learning about web development in general and a deeper knowledge of the Python programming language, the Django Framework, HTML, and CSS.

A best practice I enforced myself to follow on this project was TDD (Test Driven Development), which is about working first on writing the tests for a particular feature, then write the code to pass those tests. At first, it felt awkward to do things this way, but I kept doing it because I'm aware it's a software best practice used to produce high quality code and build more robust applications. One thing that greatly helped me (and if it helped me, I'm sure it's a common practice used under a name I don't yet know), is to figure out exactly how do you want your feature to behave under all scenarios, and then write it all out. With this, it's easier to write the tests for it, and consequently, the real code that follows. This was such a smooth process that I got super impressed! Bugs were so easy to find and fix, and I knew for sure the application would behave the way I wanted to. I'm definitely doing TDD in any opportunity I can.

Django has a pretty vast ecosystem, there's almost always a module for everything you need web related. I familiarized myself with many of the Django-way of doing things: Models, urls, project settings, quering databases, signals, CRUD operations, testing, request and response objects, static file management, user uploaded files, template language (Jinja), Django CLI commands from the project directory like shell, got familiar with the official documentation, and much more.

As for Python, since Django works with Python, I inevitably learned more about the language itself in general. I feel I improved a lot with Python Object Oriented Programming. I particularly learned a lot about the "os" and "os.path", since they were required for this project. Using Python, I learned how to remove files, work around directories, and even execute shell commands. I also got a bit familiar with the 'subprocess' module.

For HTML and CSS, I did my best to apply good practices. For example, keeping the HTML code clean, give the elements meaningful class names, use HTML semantic elements, and use mobile first approach. It's amazing how I didn't know much how to do complex things on CSS a month ago, things like Flexbox, Grid, or media queries. I got so familiar with Flexbox, that it has virtually become second nature. I also used Grid to some extent. I now feel confident working with boxes, moving them around, and making the site follow the planned layout.
